# Базы данных (QuizSlides)

Документ **«База данных и взаимодействие с ней»** ведётся по порядку. Связанные материалы: [`architecture.md`](architecture.md), [`use_cases.md`](../use_cases.md).

---

## 1. Общая информация о БД

В проекте используется **одна реляционная база данных** — **PostgreSQL**. С ней работает веб-приложение на **Django**: описание таблиц задаётся **моделями** (Python-классы), физическая схема в PostgreSQL создаётся и обновляется **миграциями**.

**Что хранится в БД по смыслу:**

- **Служебные и платформенные данные Django** — пользователи и группы (`auth`), сессии браузера, записи админки, типы контента и др. (таблицы приложений из `INSTALLED_APPS`).
- **Предметная область QuizSlides** — сессии, участники, презентации, слайды, виджеты (опрос, облако слов), голоса и связанные сущности. Это модели приложения **`core`** (в БД таблицы с префиксом `core_`).

**Что в БД обычно не хранится целиком:** двоичные файлы изображений для слайдов — в таблице остаётся **путь** к файлу (`ImageField`), сами файлы — в **медиа-каталоге** на сервере (при настроенной раздаче медиа).

Сценарии использования данных — в **`use_cases.md`**; здесь — **СУБД, схема таблиц и связи**.

## 2. Используемая СУБД

| Параметр | Значение |
|----------|----------|
| **Тип** | **PostgreSQL** (реляционная СУБД) |
| **Версия** | **18** (или та, что установлена в вашем окружении) |
| **Драйвер (адаптер) Python** | **`psycopg2-binary`** (альтернатива — пакет **`psycopg`** v3; в `requirements.txt` укажите фактически используемый вариант) |

**Подключение из Django:** в `quizslides/settings.py` в `DATABASES['default']` задаются `ENGINE` = `django.db.backends.postgresql`, а также имя БД, пользователь, пароль, хост и порт (часто через переменные `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT` и файл `.env` в корне репозитория — см. проектные настройки).

## 3. Схема базы данных

**Имя базы данных по умолчанию в настройках:** `quizslides_bd` (в вашем окружении может быть другое имя).

### 3.1. Таблицы приложения `core`

В БД создаются таблицы, соответствующие моделям Django (имена таблиц по умолчанию: `core_<имя_модели_в_нижнем_регистре>`).

| Модель (таблица) | Описание |
|------------------|----------|
| **Session** | Сессия викторины (код, статус, время, макс. участников, создатель) |
| **Member** | Участник сессии (имя, связь с сессией) |
| **Reaction** | Реакция на сессию (тип Cool / Bad, количество) |
| **Presentation** | Презентация (название, статус, связь с пользователем и сессией) |
| **Slide** | Слайд презентации (номер, картинка, статус) |
| **Widget** | Виджет на слайде |
| **WordCloud** | Облако слов (лимит слов, вопрос) |
| **AnswerOption** | Вариант ответа (номер, текст) |
| **Quiz** | Опрос / викторина (макс. время, возможность менять ответ) |
| **Vote** | Голосование (связь с участником, опросом, презентацией, виджетом) |

### 3.2. Стандартные таблицы Django

Дополнительно создаются служебные таблицы, в том числе: **`auth_user`**, **`auth_group`**, **`django_migrations`** и другие — в зависимости от `INSTALLED_APPS` (админка, сессии, типы контента и т.д.).

## 4. ER-диаграмма (описание связей)

Рисунок в документе не приводится; ниже — **ключевые связи** (кратность с точки зрения ORM):

- **Session** → **User** (создатель): **many-to-one** (`ForeignKey` на пользователя).
- **Session** → **Member**: **one-to-many**.
- **Session** → **Reaction**: **one-to-many**.
- **Session** → **Presentation**: **one-to-many**.
- **Presentation** → **Slide**: **one-to-many**.
- **Slide** → **Widget**: **one-to-many**.
- **Widget** → **Quiz**, **WordCloud**, **AnswerOption**: по одной связи **one-to-many** на каждый тип дочерней сущности (отдельные `ForeignKey` из дочерних моделей к `Widget`).
- **Vote** связывает **Member**, **Quiz**, **Presentation** и **Widget** (все связи через внешние ключи в одной строке голосования).

Точные имена полей и правила удаления (**`CASCADE`** / **`SET_NULL`**) — в `core/models.py` и в миграциях.

## 5. Взаимодействие приложения с БД

- Доступ к данным из кода — через **ORM Django** (`core/models.py`, пользователь — `auth.User`).
- Изменение схемы — команды **`makemigrations`** и **`migrate`**.
- Поле **`Slide.picture`**: в БД хранится путь к файлу, не само изображение.

---

## 6. Миграции `core`

Начальная схема домена: **`core/migrations/0001_initial.py`** (зависимость от `AUTH_USER_MODEL`).
