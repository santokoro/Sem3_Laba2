# Базы данных (QuizSlides)

Документ строится **по порядку**: сначала общая картина, далее подключение, работа приложения с БД, модели и остальное.

Связанные материалы: [`architecture.md`](architecture.md), [`use_cases.md`](../use_cases.md).

---

## 1. Общая информация о БД

В проекте используется **одна реляционная база данных** — **PostgreSQL**. С ней работает веб-приложение на **Django**: описание таблиц задаётся **моделями** (Python-классы), физическая схема в PostgreSQL создаётся и обновляется **миграциями**.

**Что хранится в БД по смыслу:**

- **Служебные и платформенные данные Django** — пользователи и группы (`auth`), сессии браузера, записи админки, типы контента и др. (таблицы приложений из `INSTALLED_APPS`).
- **Предметная область QuizSlides** — сессии презентаций, участники, презентации, слайды, виджеты (опрос, облако слов), голоса и связанные сущности. Это модели приложения **`core`** (в БД таблицы с префиксом `core_`).

**Что в БД обычно не хранится целиком:** двоичные файлы изображений для слайдов — в таблице остаётся **путь** к файлу (`ImageField`), сами файлы лежат в **медиа-каталоге** на сервере (при настроенной раздаче медиа).

Связь с продуктовыми сценариями (кто что создаёт, как идёт опрос и т.д.) — в **`use_cases.md`**; этот файл описывает **схему данных и технические детали** хранения.

## 2. СУБД и подключение

| Параметр | Значение |
|----------|----------|
| СУБД | **PostgreSQL** |
| Backend Django | `django.db.backends.postgresql` |
| Настройка | `quizslides/settings.py`, словарь `DATABASES['default']` |
| Имя БД, пользователь, пароль, хост, порт | Через переменные окружения `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT` (при наличии файла `.env` в корне проекта — подстановка из него); иначе значения по умолчанию из `settings` |

Драйвер PostgreSQL для Python нужно установить отдельно (например пакет `psycopg` или `psycopg2-binary`) — в минимальном `requirements.txt` он может отсутствовать, его добавляют при работе с БД.

## 3. Взаимодействие приложения с БД

- Доступ к данным только через **ORM Django** (модели в `core/models.py`, стандартная модель пользователя — `auth.User`).
- Схема таблиц в PostgreSQL создаётся и обновляется **миграциями**: `python manage.py makemigrations`, `python manage.py migrate`.
- Прикладные таблицы домена лежат в приложении **`core`** (префикс имён таблиц по умолчанию: `core_`).

Файлы из поля **`Slide.picture`** (`ImageField`) в БД не кладутся: в таблице хранится путь к файлу; сами изображения — в каталоге медиа на диске (см. настройки `MEDIA_*` при их включении).

## 4. Модели приложения `core` (логическая схема)

Первичные ключи у кастомных моделей — **`AutoField`** с явными именами (`id_session`, `id_presentation`, …).

### 4.1. Центральные сущности

| Модель | Назначение | Важные связи |
|--------|--------------|----------------|
| **Session** | Сессия презентации (код, лимит людей, время, статус) | `id_user` → `User` (**SET_NULL**); каскадно удаляются связанные `Member`, `Reaction`, `Presentation` |
| **Presentation** | Презентация в рамках сессии | `id_user` → `User` (**SET_NULL**); `id_session` → **Session** (**CASCADE**) |
| **Slide** | Слайд презентации | `id_presentation` → **Presentation** (**CASCADE**) |
| **Widget** | Виджет на слайде (контейнер для типа контента) | `id_slide` → **Slide** (**CASCADE**) |

### 4.2. Участники и реакции

| Модель | Назначение | Связи |
|--------|--------------|--------|
| **Member** | Участник сессии | `id_session` → **Session** (**CASCADE**) |
| **Reaction** | Реакция в сессии (`type`: Cool / Bad, `quantity`) | `id_session` → **Session** (**CASCADE**) |

### 4.3. Контент виджета: опрос и облако слов

| Модель | Назначение | Связи |
|--------|--------------|--------|
| **Quiz** | Опрос (`max_time`, `changeability`) | `id_widget` → **Widget** (**CASCADE**) |
| **AnswerOption** | Вариант ответа опроса | `id_widget` → **Widget** (**CASCADE**) |
| **WordCloud** | Облако слов (`word_limit`, `question`) | `id_widget` → **Widget** (**CASCADE**) |

### 4.4. Голосование

| Модель | Назначение | Связи |
|--------|--------------|--------|
| **Vote** | Голос участника | `id_member` → **Member**; `id_quiz` → **Quiz**; `id_presentation` → **Presentation**; `id_widget` → **Widget** (все **CASCADE**) |

Цепочка по смыслу: **User** → **Session** → **Presentation** → **Slide** → **Widget** → (**Quiz** / **WordCloud** / **AnswerOption**); **Member** привязан к **Session**; **Vote** связывает участника, квиз, презентацию и виджет.

## 5. Стандартные таблицы Django

После `migrate` в БД появляются таблицы приложений **`django.contrib.auth`**, **`sessions`**, **`admin`**, **`contenttypes`** и др. — по списку `INSTALLED_APPS`. Кастомных моделей в **`accounts/models.py`** в текущем виде нет: регистрация и пользователи опираются на **`auth.User`**.

---

## 6. Миграции `core`

Начальная схема домена: файл **`core/migrations/0001_initial.py`** (зависимость от `AUTH_USER_MODEL`).

При изменении моделей: снова `makemigrations` и `migrate`.

---

## 7. Админка и данные

Модели **`core`** зарегистрированы в Django Admin — правки записей напрямую в таблицах через `/admin/` (осторожно на продакшене: бизнес-инварианты лучше дублировать валидацией в коде).
